Detailed Notes on Python Virtual Environments

1. What is a Virtual Environment?

A virtual environment is an isolated Python runtime environment that allows a project to have its own:

* Python interpreter configuration
* Installed packages
* Dependency versions

It prevents conflicts between different Python projects on the same machine.

⸻

2. Why Virtual Environments Are Important

Dependency Isolation

Different projects often require different package versions.

Example:

* Project A → numpy==1.24
* Project B → numpy==2.0

Without isolation, installing one version may break another project.

⸻

Reproducibility

Virtual environments help other developers recreate the exact setup using:

pip freeze > requirements.txt

⸻

Cleaner System Python

Avoids polluting the global Python installation with unnecessary libraries.

⸻

Safer Experimentation

You can test packages without affecting other projects.

⸻

3. How Virtual Environments Work Internally

A virtual environment creates:

* A local Python executable
* A local site-packages directory
* Activation scripts

Structure example:

project/
│
├── venv/
│   ├── Scripts/ or bin/
│   ├── Lib/
│   └── pyvenv.cfg
│
├── app.py
└── requirements.txt

⸻

4. Types of Virtual Environment Tools

A. venv (Built-in Standard)

Recommended for most projects.

Create:

python -m venv venv

⸻

B. virtualenv

Older but faster and more customizable.

Install:

pip install virtualenv

Create:

virtualenv venv

⸻

C. conda

Popular in:

* Data science
* Machine learning
* Scientific computing

Manages:

* Python packages
* Non-Python dependencies

⸻

D. pipenv

Combines:

* Virtual environments
* Dependency management

Uses:

* Pipfile
* Pipfile.lock

⸻

E. poetry

Modern dependency management tool.

Features:

* Package management
* Dependency resolution
* Publishing support

Uses:

* pyproject.toml

⸻

5. Creating a Virtual Environment

Using venv

Windows

python -m venv venv

Linux/macOS

python3 -m venv venv

⸻

6. Activating the Environment

Windows CMD

venv\Scripts\activate

Windows PowerShell

.\venv\Scripts\Activate.ps1

Linux/macOS

source venv/bin/activate

⸻

7. Identifying an Active Environment

Terminal changes:

(venv) C:\project>

The (venv) prefix indicates activation.

⸻

8. Installing Packages

Example:

pip install flask

Installed only inside the environment.

⸻

9. Listing Installed Packages

pip list

⸻

10. Exporting Dependencies

Using requirements.txt

Generate:

pip freeze > requirements.txt

Install later:

pip install -r requirements.txt

⸻

11. Deactivating the Environment

deactivate

Returns terminal to system Python.

⸻

12. Deleting a Virtual Environment

Simply delete the environment folder:

rm -rf venv

Windows:

rmdir /s venv

⸻

13. Best Practices

A. Name Convention

Common names:

* venv
* .venv
* env

Recommended:

.venv

⸻

B. Never Upload the Environment Folder

Do NOT push:

venv/

Reason:

* Very large
* OS-dependent
* Re-creatable from requirements.txt

⸻

C. Use .gitignore

Example:

venv/
.venv/
__pycache__/
*.pyc

⸻

D. Always Include requirements.txt

This allows others to rebuild the environment.

⸻

14. Virtual Environment vs Global Python

Feature	Global Python	Virtual Environment
Package Isolation	No	Yes
Safe for Projects	No	Yes
Dependency Conflicts	Common	Rare
Recommended	No	Yes

⸻

15. Common Commands Cheat Sheet

Task	Command
Create env	python -m venv venv
Activate (Windows)	venv\Scripts\activate
Activate (Linux/macOS)	source venv/bin/activate
Install package	pip install package_name
Export packages	pip freeze > requirements.txt
Install from file	pip install -r requirements.txt
Deactivate	deactivate

⸻

16. Common Problems and Solutions

Problem: Activation Script Disabled (PowerShell)

Solution:

Set-ExecutionPolicy RemoteSigned -Scope CurrentUser

⸻

Problem: Wrong Python Version

Check:

python --version

Specify version:

python3.11 -m venv venv

⸻

Problem: Packages Installing Globally

Check active environment:

which python

Windows:

where python

⸻

19. Which Format Should You Upload to GitHub?

Recommended Format for GitHub

Upload:

✔ Source code
✔ requirements.txt
✔ README.md
✔ .gitignore
✔ configuration files

Do NOT upload:

✘ venv/
✘ .venv/
✘ __pycache__/
✘ large datasets (unless required)
